# ADR-008: Web 検索によるナレッジ取り込み（参考情報の自動収集）

## ステータス
proposed

## コンテキスト
確認支援・入力支援の判定品質は、参考情報（`references_master` などの RAG 知識源）の充実度と鮮度に依存する。NIST / CIS / AWS ドキュメント / OWASP 等のベストプラクティスを人手だけで整備・更新するのは負荷が高く、陳腐化しやすい。インターネットから自動的に収集・要約して取り込みたい。

一方、本システムの従来方針は「**外部送信は基本なし**」（Bedrock は AWS 信頼境界内で完結し、Anthropic/OpenAI に届かない）。Web 検索・URL 取得は **第三者（検索プロバイダ・取得先サイト）にクエリやデータが渡る新たな外部接点**であり、リスク評価が要る。母体 application-form-poc は Tavily Web Search で参考情報リフレッシュ/取り込みを実装した実績がある（ADR backend/003・004）。

## 決定
1. **主方式はバッチ取り込み（admin 承認フロー経由）。**
   - Web 検索/クロール → 本文抽出 → Claude が要約・分類 → **提案** → admin が承認 → `references_master`（合成事例も可）へ反映。
   - 確認支援・入力支援の**実行時は DB（RAG）を引くだけ**で、Web へは出ない。再現性・レイテンシ・コスト・監査の面で有利。
2. **実行時の動的 Web 検索はフォールバック限定。**
   - 「DB にヒットが無い」「時事性が高いキーワード」のときだけ補完検索し、**出典付き・`confidence: low`** で提示。結果は後で admin がナレッジ化する。
3. **実装は 2 系統を候補とし、[調査結果](../research/web-search-options.md) を踏まえて選定する。**
   - (A) 検索 API（Tavily / Brave / Exa 等）+ 本文抽出 + Claude 要約（柔軟・母体実績あり）
   - (B) **Amazon Bedrock Knowledge Bases の Web Crawler データソース**（信頼ソースの URL を登録 → 自動クロール → KB 化。マネージドで運用が軽く、`bedrock_kb` 戦略（C5）と直結）

## 検討した代替案
- **実行時に毎回 Web 検索**: 鮮度は最高だが、非決定性（同じ申請で結果が変わる）・レイテンシ増・コスト増・リスク増。→ フォールバックに限定。
- **手動整備のみ**（Web 連携しない）: 安全だが負荷大・陳腐化。→ 不採用。

## 結果・リスクと対策
新たな外部接点が増えるため、以下を必須とする。

| リスク | 対策 |
|---|---|
| 外部への情報漏えい（PII・社外秘がクエリに混入） | 検索クエリは **PII フィルタ後のキーワードのみ**。回答本文をそのまま送らない |
| プロンプトインジェクション（取得ページの悪意ある指示） | 取得本文は「事実抽出・要約」に限定し命令を無視するプロンプト設計。生本文を判定プロンプトへ直接流さない |
| SSRF（任意 URL 取得） | private IP / link-local / メタデータ IP を弾く（母体 R6 相当）、リダイレクト制御 |
| API キー管理 | Secrets Manager 保管・ローテーション。コミット禁止 |
| 監査・再現性 | 取得 URL・取得日時・出典を保存。判定根拠に出典を併記。低信頼度フラグ |
| コスト・レート制限 | バッチ主体でレート/課金を平準化。実行時検索は条件付きで抑制 |

- 外部接点が増えるので、セキュリティ整理（外部送信点の一覧）を別途更新する。
- 取り込みは確認支援とは**非同期**（admin 運用）なので、実行時の信頼境界モデルは大きく変えない。
